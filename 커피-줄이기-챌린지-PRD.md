## 2. 1페이지 요약

### 문제

커피를 줄이고 싶어도 "얼마가 줄었는지"가 눈에 안 보이면 금방 흐지부지돼요. 기록이 귀찮으면 리텐션이 급격히 떨어져요.

### 가치

커피를 끊게 만드는 앱이 아니라, 사용자가 설정한 기준과 실제 기록을 비교해 **절감액(기록 기반 추정)** 을 보여주는 기록 도구예요. 원탭 입력으로 "기록 → 절감 확인" 루프를 10초 안에 끝낼 수 있어요.

### 대상 사용자

반복적으로 커피를 사는 직장인/학생 중 특정 항목(커피)만 관리하고 싶은 사용자예요.

### 경쟁 포지셔닝

글로벌 카페인 추적 앱(HiCoffee, RECaf 등)은 mg 단위 정밀 추적에 집중하지만 감량 여정 설계가 없고, 한국 습관 챌린지 앱(챌린저스, 마이루틴 등)은 범용 체크박스만 제공해요. **"카페인 감량 목표 + 간편 잔 수 추적 + 게이미피케이션"을 결합한 앱은 사실상 부재**하며, 토스 생태계에서 로그인 없이 즉시 시작할 수 있는 접근성이 핵심 차별점이에요.

### 핵심 기능 (MVP)

1. 오늘 커피 원탭 기록(+1/−1, 기본 단가 제공/수정)
2. 목표 설정(주 n회 이하 또는 월 예산) + 기준(평소 평균) 설정
3. 오늘/이번 달 절감액 및 절감 예상치(기록 기반 추정)
4. 달력/통계(월 요약, 기록일/절감일 표시)
5. 프리미엄(1회 구매): 고급 리포트, PDF 내보내기, 대체 루틴 템플릿

### 수익화

프리미엄 **소모성 IAP**(1회 구매, 기간제 이용권). "30일 이용권 / 365일 이용권" 형태로 판매하며, 구매 시 즉시 프리미엄 기간이 지급돼요.

### 핵심 제약(앱인토스)

- 건강/의학 조언, 치료·진단 표현을 사용하지 않아요.
- "절감 보장"처럼 확정적 표현을 금지하고 "기록 기반 추정/예상"으로만 안내해요.
- **IAP 구독 불가** → 소모성 아이템(기간제 이용권) 1회 구매로 대체해요.
- 결제는 강제 유도 없이 사용자가 스스로 선택하게 해요(동등한 무료 사용 경로 제공).
- 라이트 모드 고정, 핀치줌 비활성화 필수.
- MVP는 동일 기기 저장 기준으로 운영해요. 재설치·기기 변경 시 데이터/IAP 복원은 Phase 2에서 서버 동기화와 함께 다뤄요.

## 3. 목표와 성공지표

### 목표

- 2주 내 출시 가능한 MVP 완성(WebView + 네이티브 저장소 + 핵심 루프)
- 사용자가 "오늘 기록 → 절감 확인"을 1분 내 완료

### KPI

| 카테고리 | 지표 | 목표 |
|---|---|---|
| 활성/리텐션 | D1, D7 리텐션 | D1 ≥ 40%, D7 ≥ 20% |
| 활성/리텐션 | 1인당 주간 기록 일수 | 주 3일 이상 비율 ≥ 30% |
| 핵심 행동 | 온보딩 완료율 | ≥ 70% |
| 핵심 행동 | 목표/기준 설정 완료율 | ≥ 50% |
| 핵심 행동 | 첫 기록 생성 전환율 | ≥ 60% |
| 수익 | 프리미엄 화면 진입 → 구매 전환율 | 측정 후 설정 |
| 수익 | 결제 실패율 | < 5% |
| 가드레일 | "절감액 부정확" 피드백 비율 | < 3% |
| 가드레일 | 결제/환불 문의율 | < 2% |

### 분석 이벤트 설계 개요

온보딩 단계별 이탈, 목표 유형(주/월)과 리텐션 상관, 절감액 노출 지점별 재방문 영향, 결제 퍼널(노출→구매→복원)을 추적해요.

## 4. 범위

### MVP 범위 (In)

- 온보딩 3스크린
- 오늘 커피 원탭 기록(수량/단가)
- 목표 설정(주 n회 이하 또는 월 예산)
- 기준(평소 평균) 설정
- 절감액/절감 예상치 계산 및 표시(기록 기반 추정)
- 달력/월 요약 통계
- 네이티브 저장소(오프라인 우선, 브라우저 fallback 허용)
- 앱인토스 런타임에서 토스 로그인 진입(appLogin)
- 프리미엄(IAP 소모성): 고급 리포트, PDF 내보내기(saveBase64Data), 대체 루틴 템플릿

### 비범위 (Out)

- 카드/계좌 지출 자동 연동(오해/검수 리스크)
- 건강/카페인 관련 코칭/진단
- 포인트/리워드 지급(어뷰징 리스크)
- 재설치/기기 변경 시 데이터 및 프리미엄 복원
- 완료 주문 이력 기반 구매 복원
- 간식 관리(Phase 2)
- 인앱 광고(MVP 미포함)
- mTLS 서버(IAP 주문 조회 API) — MVP 후속 업데이트
- PDF/리포트 시스템 공유 — Phase 2

### Phase 2 아이디어 (Backlog)

- 간식 항목 추가(커피와 동일 구조)
- mTLS 서버 구축(IAP 주문 상태 서버 검증)
- 서버 동기화/백업
- 스마트 발송(앱인토스 푸시 SDK)
- 공유 카드 자동 생성(이미지 템플릿) + 공유 리워드
- 개인화 인사이트(요일/상황별 패턴)
- CSV 내보내기(프리미엄)
- PDF/리포트 시스템 공유

## 5. 사용자 & 시나리오

### 핵심 페르소나

**1) 직장인 절약러** — 점심 후 습관처럼 커피를 사요. 줄이고 싶지만 기록은 귀찮아요. 원하는 것: 빠른 입력, 숫자로 보이는 변화. 기존 카페인 추적 앱(HiCoffee 등)은 mg 단위 입력이 번거로워서 포기한 경험이 있어요.

**2) 소비 다이어트 입문자** — 가계부는 부담스러워요. 특정 항목만 줄이고 싶어요. 원하는 것: 작은 성공 경험(절감액/달성률). 챌린저스 같은 예치금 모델은 부담스러워요.

**3) 학생/취준생** — 예산이 빡빡해요. 커피 지출을 관리하고 싶어요. 원하는 것: 월 예산 기반 관리, 저장/공유 가능한 결과.

### 핵심 유저 여정

**신규 진입**
1. 앱 진입 → 온보딩 3스크린(숫자 임팩트 → 오해 해소 → 결과 미리보기)
2. [챌린지 시작하기]를 누르면 로그인/식별값 설정으로 가고, [나중에]를 누르면 기본값으로 바로 시작해요
3. 목표/기준 빠른 설정
4. 홈에서 원탭 기록
5. 오늘 절감액/이번 달 누적 절감액 확인

**반복 사용**
1. 매일 홈에서 원탭 기록
2. 달력에서 기록일/절감일과 월 요약 확인
3. 필요할 때 목표와 기준을 다시 설정

**수익화**
1. 한 달 리포트/요약에서 "더 자세히 보고 저장하고 싶다" 니즈 발생
2. 프리미엄 화면 → 이용권 구매(소모성 아이템)
3. 만료 후 재구매 또는 무료로 계속 사용

## 6. 정책/검수/컴플라이언스 (앱인토스 전용)

### 앱 유형 판정

비게임(기록/통계 기반 소비 습관 도구)이에요.

### 서비스 오픈 정책 관련 체크

- 제한 서비스/자격 필요 카테고리에 해당하지 않아요(금융 상품 중개·투자 자문 아님).
- 금융/계좌/카드 내역을 직접 조회하거나 자동 연동하지 않아요.
- 절감액은 정보성 지표로만 제공하고, 투자/재무 조언처럼 해석될 수 있는 문구를 피해요.
- 어뷰징 방지: 동일 기능 반복 출시가 아닌 단독 서비스예요.

### 외부 링크/자사 앱 설치 유도

- 외부 링크를 사용하지 않아요.
- 개인정보처리방침, 이용약관은 앱 내 텍스트 화면으로 제공해요.
- MVP에서 리포트 공유는 시스템 공유(텍스트 중심)를 사용하고, 공유 링크가 자사 웹으로 랜딩되지 않아요.
- PDF 내보내기는 saveBase64Data SDK로 기기에 직접 저장하며 외부 서버를 거치지 않아요.

### 다크패턴 방지 정책 준수

| 정책 항목 | 준수 방안 |
|---|---|
| 1. 진입 시 바텀시트 차단 금지 | 온보딩은 풀스크린 페이지로 구성하고, 진입 직후 바텀시트/팝업을 띄우지 않아요 |
| 2. 뒤로가기 시 인터럽트 금지 | 뒤로가기 시 이전 화면으로 즉시 이동해요. 이탈 방지 바텀시트를 사용하지 않아요 |
| 3. 나갈 수 있는 선택지 제공 | 프리미엄 화면에 [무료로 계속 쓰기]를 동등한 크기·위치로 배치해요 |
| 4. 예상치 못한 광고 금지 | MVP에서 인앱 광고를 사용하지 않아요 |
| 5. CTA 명확성 | 모든 CTA는 다음 행동을 예측할 수 있게 작성해요(예: "저장하기", "30일 이용권 구매하기") |

### 기술 제약 체크

| 제약 | 대응 |
|---|---|
| 라이트 모드 고정 | 다크모드 미지원. 테마를 라이트 모드로 고정하고 내비게이션 바도 라이트 모드 설정 |
| 핀치줌 비활성화 | meta viewport에 `initial-scale=1, maximum-scale=1, user-scalable=no` 적용 |
| IAP 구독 불가 | 소모성 아이템(기간제 이용권)으로 대체 |
| 기기 변경 시 데이터 유지 | MVP 범위에서 보장하지 않아요. 동일 기기 내 userKey 분리만 적용하고, 재설치·기기 변경 복원은 Phase 2에서 다뤄요 |
| iframe 사용 금지 | 모든 콘텐츠를 미니앱 내부에서 직접 렌더링 |
| 자사 로그인 금지 | 토스 로그인만 사용 가능 |
| 내비게이션 바 커스텀 제한 | 크기/색상/위치 변경 불가, 홈 버튼·기능 버튼만 선택적 추가 |

### S04 로그인/목표 설정

**목적:** 필요할 때 토스 로그인을 시도하고, 현재 기기에서 사용할 계산 기준을 설정해요.

**로그인 플로우**
1. 서비스 소개(온보딩)를 거친 뒤에만 로그인 진입을 보여줘요.
2. 앱인토스 런타임이면 appLogin SDK를 호출해 로그인 성공 여부를 확인해요.
3. userKey 자동 획득 경로가 준비되지 않았거나 실패하면, 현재 기기에서 사용할 식별값을 직접 입력해 계속 진행할 수 있어요.
4. 로그인 화면에서 '닫기' → 이전 화면(S03)으로 복귀.
5. 동일 기기 안에서는 active userKey 기준으로 데이터가 분리돼요. 재설치·기기 변경 후 자동 복원은 Phase 2 범위예요.

**목표 설정 UI**
- 목표 유형: 주 n회 이하 / 월 예산
- 기준(평소 평균): "평소 하루 커피 몇 잔이에요?", "평균 단가(원)"
- [저장하기]

**입력/검증**

| 필드 | 범위 | 에러 문구 |
|---|---|---|
| 주간 횟수 | 0–50 | "횟수는 0 이상으로 입력해요." |
| 월 예산 | 0–5,000,000원 | "금액은 0원 이상으로 입력해요." |
| 기준 잔수 | 0–20 | "잔수는 0 이상으로 입력해요." |
| 단가 | 100–50,000원 | "단가는 100원 이상으로 입력해요." |

**기준 변경 정책:** 변경한 날짜부터 새 기준이 적용돼요. 과거 날짜 계산은 과거 기준을 유지해요.

## 10. 데이터 & 이벤트(Analytics)

### 데이터 모델

**Entry**
- id, userKey, date(YYYY-MM-DD), coffee_count(정수), unit_amount(원)

**Goal**
- userKey, goal_type(weekly_limit | monthly_budget), weekly_limit?, monthly_budget?

**BaselineVersion**
- userKey, effective_from(YYYY-MM-DD), avg_per_day, unit_amount

**PremiumStatus**
- userKey, is_premium(boolean), plan(30d | 365d), purchased_at, expires_at, last_order_id

### 저장소 키 설계 (현재 기기 기준 분리)

네이티브 저장소(Storage SDK) 또는 브라우저 fallback 저장소의 키는 모두 `{userKey}:` prefix를 사용해요.

| 키 패턴 | 설명 |
|---|---|
| `{userKey}:entries:{date}` | 해당 날짜의 기록 |
| `{userKey}:goal` | 현재 목표 설정 |
| `{userKey}:baselines` | 기준 버전 히스토리 |
| `{userKey}:premium` | 프리미엄 상태 |

동일 기기 안에서 다른 userKey로 전환하면 새 prefix의 빈 데이터로 시작해요. 같은 userKey를 다시 선택하면 해당 기기에 남아 있는 데이터를 다시 불러와요. 새 기기/재설치 복원은 MVP 범위에 포함하지 않아요.

### 핵심 계산 규칙

```
expected_daily = baseline.avg_per_day × baseline.unit_amount
actual_daily   = coffee_count × unit_amount
saving_daily   = max(0, expected_daily − actual_daily)   // 기록 기반 추정
overspend_daily = max(0, actual_daily − expected_daily)
```

기간 합산은 "기록이 있는 날짜"만 포함해요. 미기록일은 별도 표기해요.

## 11. 기술/아키텍처

### 구현 방식

| 항목 | 선택 |
|---|---|
| 클라이언트 | WebView(반응형) — Granite 프레임워크 |
| 저장 | 네이티브 저장소(Storage SDK) 우선 + 브라우저 fallback — 동일 기기 기준 userKey 분리 |
| 서버 | **MVP: 서버 없음** — 클라이언트 SDK만으로 출시(Q4 확정). Phase 2에서 IAP 주문 조회 서버 추가 |
| 프레임워크 | Vite + React (또는 Vue) |

### 로그인/식별

- MVP는 `guest_local` 같은 현재 기기용 기본 식별값으로도 시작할 수 있어요.
- 앱인토스 런타임에서는 appLogin SDK 진입을 제공해요.
- userKey 자동 획득은 별도 교환 경로가 준비된 환경에서만 사용해요. 준비되지 않으면 현재 기기용 식별값 입력으로 계속 진행해요.
- 저장소 키는 active userKey prefix로 분리해 동일 기기 안에서 사용자 데이터를 구분해요.
- 재설치·기기 변경 후 자동 복원은 Phase 2 범위예요.

### 결제(프리미엄 IAP)

**⚠️ v1.1 변경: 소모성 아이템으로 전환 권장**

v1.0에서는 "비소모성 아이템 + 기간 만료"로 설계했으나, 앱인토스 IAP 정의상 비소모성 = 영구 소유이므로 기간 만료와 모순돼요. **소모성 아이템으로 전환해 "구매 즉시 n일 프리미엄 지급 → 소진"** 방식이 적합해요. 채널톡으로 최종 확인 후 콘솔 등록을 진행해요.

**상품 구성(콘솔 등록)**

| SKU | 유형 | 가격 | 설명 |
|---|---|---|---|
| premium_30d | 소모성 | 1,900원 | 프리미엄 30일 이용권 (구매 즉시 소진, 30일 프리미엄 지급) |
| premium_365d | 소모성 | 14,900원 | 프리미엄 365일 이용권 (구매 즉시 소진, 365일 프리미엄 지급) |

**구매 플로우**
1. getProductItemList()로 상품 목록 조회
2. 사용자가 이용권 선택
3. createOneTimePurchaseOrder() 호출 → Apple/Google 결제창
4. 성공 시: completeProductGrant() 호출(소모성 아이템 소진 완료) → PremiumStatus 갱신(expires_at = now + 기간) → 리포트/내보내기/템플릿 잠금 해제
5. 실패 시: Toast "결제에 실패했어요. 다시 시도해요." + 사유 안내

**복원/미결 주문 처리**
1. 앱 재진입 시 getPendingOrders()로 미결 주문을 확인해요.
2. 미결 주문이 있으면 → 상품 지급(PremiumStatus 갱신) → completeProductGrant() 호출로 마무리해요.
3. 설정 > "구매 복원" 탭은 완료 주문 전체 복원이 아니라 **미결 주문 재처리** 용도로 제공해요.
4. 완료 주문 이력 기반 복원과 새 기기 복원은 Phase 2에서 서버 검증과 함께 다뤄요.

**만료 처리**
- 앱 진입 시 PremiumStatus.expires_at 확인
- 만료된 경우: 프리미엄 기능 잠금, "이용권이 만료됐어요. 다시 구매할 수 있어요." 안내
- 무료 기능(기록/달력/기본 통계)은 그대로 유지

**소모성 vs 비소모성 비교 (전환 사유)**

| 항목 | 비소모성(v1.0) | 소모성(v1.2 현실안) |
|---|---|---|
| 정의 | 영구 소유 | 구매 후 소진 |
| 기간 만료와의 정합성 | ❌ 모순 (영구인데 만료) | ✅ 적합 (소진 후 기간 지급) |
| 복원 | getCompletedOrRefundedOrders로 이력 조회 가능 | 소진 완료 후에는 이력 조회로 재지급 불가 |
| 기기 변경 대응 | 주문 조회로 복원 가능 | MVP 미지원 (동일 기기 미결 주문 처리만 지원) |
| Phase 2 보강 | — | 주문 상태 서버 검증 + 동기화 저장소 추가 |

### 보안/개인정보

- 개인정보 최소화: 식별값(userKey 또는 현재 기기용 식별값)과 커피 기록만 저장해요.
- 로그/에러 수집 시 개인 입력값(메모 등)은 마스킹하거나 수집하지 않아요.
- 데이터 초기화는 2단계 확인으로 실수 삭제를 방지해요.
- MVP에서는 서버가 없으므로 mTLS 불필요. Phase 2에서 IAP 주문 조회 서버 추가 시 mTLS 적용.

### PDF 내보내기 (Q3 확정)

- saveBase64Data SDK로 기기에 직접 저장해요.
- 클라이언트에서 HTML → Canvas → PDF(또는 직접 PDF 라이브러리) → Base64 변환 후 SDK 호출.
- 시스템 공유(다른 앱으로 보내기)는 Phase 2에서 추가해요.
- 저장 성공/실패 각각 Toast로 안내해요.

## 13. 배포/출시/운영 계획

### 배포 절차

1. 개발 빌드로 기능 점검(온보딩/로그인/기록/달력/설정/결제)
2. 샌드박스 앱에서 IAP 테스트(성공/실패/복원 3종 필수)
3. saveBase64Data(PDF) 실기기 테스트
4. 토스앱 테스트 스킴으로 실기기 확인
5. 번들(.ait) 생성 및 업로드
6. 검토 요청 → 수정 대응 → 승인
7. 출시 후 모니터링 및 핫픽스/롤백 준비

### 버전 관리/롤백/핫픽스

**치명 이슈 우선순위**
1. 기록 저장 불가
2. IAP 결제/복원 오류
3. 절감 계산 대규모 오표시

롤백 시나리오: 이전 번들로 즉시 복귀 가능하도록 릴리즈 단위 분리.

### 운영/모니터링

| 카테고리 | 모니터링 항목 |
|---|---|
| 에러 | 저장 실패율, 화면 로드 실패율 |
| 성능 | 첫 로드 시간, 달력 렌더링 시간 |
| 결제 | 결제 실패율, 미결 주문 처리 성공률, 이용권 구매 전환율 |
| 메모리/데이터 | 메모리 누수, 과도한 데이터 사용 |
| 피드백 | 콘솔 고객센터 정보와 외부 문의 채널 운영 |

### 고객센터/문의/신고 대응

- 콘솔에 고객센터 이메일/연락처 등록 필수.
- 문의 유형 분류: 기능 오류 / 계산 문의 / 결제·환불 / 피드백.
- 결제 문의는 "미결 주문 재처리" 안내를 우선 제공하고, 해결이 안 되면 고객센터 경로를 안내해요.
- 재설치·기기 변경 복원은 MVP 미지원 범위임을 명확히 안내해요.
- IAP 환불은 Apple/Google 정책을 따르므로, 환불 절차를 안내해요.

## 14. 리스크 & 오픈 이슈

### 정책 리스크

| 리스크 | 영향 | 대응 |
|---|---|---|
| "통장/잔액" 표현 | 자동 연동 오해 → 검수 반려 | "지출 변화/기록 결과"로 통일(확정) |
| ROI 압박 카피 | 과도한 결제 유도 → 검수 반려 | 삭제 완료, 정보성 안내로만 표현 |
| IAP 소모성/비소모성 선택 | 잘못된 유형 등록 시 검수 반려 | 소모성 아이템 전환 권장, 채널톡 최종 확인(Q1) |

### 제품 리스크

| 리스크 | 대응 |
|---|---|
| 기록 안 하면 가치가 안 보여요 | 빈상태에서 행동 유도 안내 강화 |
| 기준 설정이 어려워요 | 온보딩 후 "추천 기본값" 제공 + 언제든 수정 안내 |
| 재설치·기기 변경 복원을 기대할 수 있어요 | MVP 범위를 앱 소개/설정에 명확히 쓰고, Phase 2에서 서버 동기화 검토 |
| 재구매 전환율이 낮을 수 있어요 | 만료 전 알림(Phase 2 푸시), 365일 이용권 할인으로 LTV 확보 |

### 기술 리스크

| 리스크 | 대응 |
|---|---|
| 네이티브 저장소 용량 제한 | PDF 내보내기(saveBase64Data)로 "내 기록 보관" 제공 |
| 소모성 IAP 복원 한계 | MVP에서는 미결 주문 처리만 지원하고, 완료 주문/새 기기 복원은 Phase 2에서 서버 검증 보강 |
| userKey 자동 획득 불안정 | appLogin은 시도하되, 교환 경로가 없으면 직접 입력 fallback 허용 |
| PDF 생성 성능 | 클라이언트 PDF 라이브러리 선정 시 번들 크기/메모리 영향 확인 |
| 서버 없는 구조의 보안 | 민감 데이터 없음(userKey + 커피 기록뿐). 결제 검증은 SDK 레벨에서 처리 |

### 오픈 질문 (사용자 확인 필요)

> 아래 항목은 출시 범위를 확정하기 전에 확인이 필요해요.

**Q1.** 앱인토스 IAP에서 소모성 아이템을 "기간제 이용권"(30일/365일 프리미엄 지급 후 소진)으로 등록해도 검수에 문제가 없는지 채널톡 확인이 필요해요.

**Q2.** 기기 변경/재설치 복원이 없는 상태로 MVP를 검수 요청할 수 있는지 정책 확인이 필요해요. 필수라면 Phase 2가 아니라 출시 전 선행 과제로 승격해야 해요.

### 이번 버전에서 확정한 범위

| 항목 | 결정 | 근거 |
|---|---|---|
| PDF 내보내기 방식 | saveBase64Data SDK(기기 직접 저장) | 외부 서버 불필요, 시스템 공유는 Phase 2 |
| mTLS 서버 | MVP는 서버 없이 SDK만 출시 | 2주 일정 내 서버 구축 리스크 제거. Phase 2에서 Fly.io/Vercel 검토 |
| 로그인/저장소 | 동일 기기 기준 userKey 분리만 MVP 범위 | 현재 구조로 실현 가능하고, 새 기기 복원은 별도 동기화가 필요 |

## 15. 부록

### 용어집

| 용어 | 설명 |
|---|---|
| 기준(평소 평균) | 사용자가 설정한 "평소 커피 소비 수준"이에요 |
| 절감액 | 기준 대비 실제 기록이 낮을 때의 차이를 기록 기준으로 추정한 값이에요 |
| 미기록 | 해당 날짜에 기록이 없어 절감/초과 지출을 계산하지 않은 상태예요 |
| 소모성 아이템 | 한 번 구매하면 즉시 소진되는 IAP 상품이에요. 본 앱에서는 "구매 시 n일 프리미엄 지급 후 소진"으로 운용해요 |
| 비소모성 아이템 | 한 번 구매하면 영구 소유하는 IAP 상품이에요. 기간 만료와 모순되므로 본 앱에서는 사용하지 않아요(v1.1) |
| userKey | 토스 로그인 또는 현재 기기용 식별값으로 사용하는 사용자 식별자예요. 네이티브 저장소 키의 prefix로 사용해 데이터를 격리해요 |
| 네이티브 저장소 | 앱인토스 Storage SDK를 통해 기기에 데이터를 저장하는 방식이에요(WebView localStorage보다 안정적) |
| saveBase64Data | 앱인토스 SDK에서 제공하는 Base64 데이터를 기기에 파일로 저장하는 API예요 |
| mTLS | 앱인토스 API 호출 시 필요한 상호 TLS 인증 방식이에요. MVP에서는 서버 미사용으로 해당 없음 |
